What is Neon?

Neon is a framework for deploying classes that need to use Jini services, without having to write infrastructure code like Discovery for each and every class that needs to utilise Jini services. To attain this, Neon provides an agent framework [1]
The basic premise is to provide a Jini service for hosting objects; for instance, these objects could be business objects, which need to use jini services to handle some of their core needs, but making the business object itself into a service seems overkill, yet this object still has to exist and be managed somewhere.
Neon provides a way to make your objects collaborate safely, and to work seamlessly across your network, thus providing an application grid fabric.

How does it work?

Out of the box, Neon provides a number of agents that can be deployed at startup, that handle obtaining services, distributed transaction management, Agent checkpointing, recovery, transferring agents between different hosts, and being informed of external events. By interacting with these agents it is easy to build agents that provide business functionality, as well as other infrastructure and tool type agents, whilst still getting the most out of the Jini infrastructure; by running multiple instances of Neon on your network, each with different profiles (different CPU's, access to differ net third party libraries etc.), a grid 'landscape' is created, where the agents collaborate regardless of where they are in relation to each other
